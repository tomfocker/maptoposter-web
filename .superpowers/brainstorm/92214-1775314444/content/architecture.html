2. 架构方案选择

考虑到项目核心是 Python 及其地理处理库 (OSMnx)，哪种架构更符合你的需求？

A

单体轻量级架构 (Flask/FastAPI + Jinja2)

优点： 部署极快，镜像体积小，代码复杂度低。适合个人使用或轻量级分享。

缺点： 前端交互相对传统，复杂状态管理较难。

B

现代前后端分离 (FastAPI + React/Vue)

优点： 交互极其流畅（如搜索联想、动态进度条）。扩展性强，适合打造成一个完整产品。

缺点： Docker 镜像构建更复杂（需编译 Node.js），运行时资源占用略高。

技术权衡

方案 A (推荐)

  • 直接复用现有 Python 环境
  • 由于生成地图是耗时操作，简单架构更易处理阻塞
  • 维护成本极低

方案 B

  • 虽然体验更好，但会增加开发和维护周期
  • 对于“开箱即用”的 Docker 镜像，A 方案更容易做到稳定
